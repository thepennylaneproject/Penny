## Title

Ingest edge function has no caller authentication

## Description

The `ingest-audit` Supabase Edge Function creates a `supabase-js` client with `SUPABASE_SERVICE_ROLE_KEY` and upserts `audit_runs`, `findings`, and `orchestration_events` from the request body. There is no verification of a shared secret, Supabase JWT, or other caller credential in the handler. Attack scenario: anyone who can invoke the deployed function URL (URL discovery, leaked project ref, or browser/CORS context if exposed) sends a crafted JSON body and overwrites or poisons audit results and findings for arbitrary `project_id` values, corrupting compliance data and downstream repair/orchestration decisions.

## Impact

Full integrity loss for audit data tied to this pipeline: fake findings, deleted accountability, and wrong repair triggers for projects.

## Suggested fix

Require a server-only secret (compare timing-safe), or verify Supabase JWT with role checks, or disable public invocation and rely on Supabase function JWT verification in project config—ensure code and platform config agree.

**Affected files:** `supabase/functions/ingest-audit/index.ts`

## Proof hooks

- **[code_ref]** Handler uses service role and writes without auth check.
- **[repro_steps]** Unauthenticated POST path to ingest.
